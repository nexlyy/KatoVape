# KatoVape: план системы (бот, ассортимент, бронь, рассылки, оплата, админка)

Документ описывает, как всё связать. Пишу как план перед сборкой: что за компоненты,
какие данные ходят, какие потоки, как хостить на VPS mcr. Код по этому плану делаем
следующим этапом.

## 1. Компоненты

- **Мини-апп** (готово) — витрина внутри Telegram, 4 стиля, общий `shared/core.js`.
  Внутри Telegram человек авторизован сам по `initData`.
- **Сайт** (готово) — та же витрина в браузере, вход по логину/почте/телефону или Telegram.
- **Бот** (делаем) — хостит мини-апп (кнопка-меню), шлёт уведомления и рассылки,
  принимает бронь и заказы.
- **Backend API** (делаем) — единая точка для сайта, мини-аппа и админки. На демо это
  локальный `server/` (node:sqlite). В прод переносим на Postgres (Supabase или свой на VPS).
- **Google Sheets** — заказчик ведёт ассортимент. Синк тянет остатки в БД.
- **Админка** (делаем) — веб-панель для своих: спрос, заказы, данные клиентов.
- **Stripe** (позже) — оплата на сайте и в мини-аппе.

```
Google Sheets ──sync──> Backend/DB <──── Сайт / Мини-апп
                           │  ▲                 │
                        Бот │  │ заказы, бронь   │
                           ▼  │                 ▼
                      Клиент в TG            Админка (свои)
                                                 │
                                              Stripe
```

## 2. Модель данных (Postgres)

- `users` — уже есть в демо: id, username, email, phone, telegram_id, telegram_username,
  display_name, avatar, created_at.
- `products` — зеркало Google Sheets: id, city, category, name, brand, price, tiers(jsonb),
  nic, updated_at, синк перезаписывает.
- `flavors` — id, product_id, name, qty (или хранить как jsonb в products, как в products.json).
- `reservations` — id, user_id, product_id, flavor, city, created_at, notified_at(nullable),
  status (waiting/notified/cancelled). Для «БРОНЬ, оповести когда появится».
- `orders` — id, user_id, city, items(jsonb), sum, delivery(method+addr), status
  (new/paid/done/cancelled), stripe_session_id(nullable), created_at.
- `broadcasts` — id, author_admin, text, audience, sent_count, created_at.
- `bot_users` — telegram_id, opted_in(bool), lang, first_seen. Кто нажал /start, кому можно слать.
- `admins` — telegram_id, role (owner/manager), добавляется вручную.

RLS/доступ: клиент видит только своё (свой профиль, свои заказы, свою бронь).
Админ-эндпоинты закрыты проверкой telegram_id в `admins` (и отдельным JWT для веб-админки).

## 3. Потоки

### 3.1 Ассортимент из Google Sheets
- Заказчик ведёт таблицу: колонки `city, category, id, name, brand, price, tier1_q, tier1_p,
  tier3_p, tier5_p, flavor, qty, nic, added`. Одна строка = один вкус позиции (или лист на город).
- Синк: Google Sheets API (сервис-аккаунт, только чтение) раз в N минут или по кнопке в
  админке. Пишет в `products`/`flavors`, ставит `updated_at`. Витрина читает БД, не Sheets
  напрямую (Sheets медленный и лимитирован).
- Демо-режим: остаётся `data/*.json`, синк подменяет источник позже.

### 3.2 Бронь и уведомление о поступлении
- На сайте/в аппе у товара, которого нет в наличии, кнопка **БРОНЬ**. Жмёт залогиненный
  человек. Пишем строку в `reservations` (user_id + product + flavor + city, status=waiting).
- Синк ассортимента после обновления проверяет: появились ли позиции с qty>0, на которые
  есть `waiting`-бронь. Если да, бот шлёт этому user (по telegram_id) сообщение
  «<товар> <вкус> появился в <город>, забронировать?» с кнопкой, ставит status=notified.
- Если у человека нет telegram_id (регистрировался почтой), шлём на почту или показываем
  бейдж в профиле при следующем заходе.

### 3.3 Рассылки
- Админ в панели пишет текст (новости, новинки, акции), выбирает аудиторию (все opted_in,
  или по городу). Бэкенд ставит задачу, бот шлёт пачками с троттлингом (Telegram лимит
  ~30 msg/s), пишет `sent_count`. Только тем, кто нажал /start (`bot_users.opted_in`).

### 3.4 Заказы и оплата Stripe
- Оформление заказа уже собирает состав, доставку, сумму (`orderText` в core.js).
- Прод-вариант: заказ уходит в `orders` (status=new), клиент видит его в профиле.
- Stripe: создаём Checkout Session (сумма из заказа), возвращаем ссылку. В мини-аппе через
  `openInvoice`/внешний браузер, на сайте redirect. После оплаты Stripe шлёт webhook
  (подпись проверяем секретом), заказ → status=paid, бот пишет клиенту и в чат менеджера.
- Пока Stripe не подключён, оформление как сейчас: открывает чат менеджера с готовым текстом.

### 3.5 Админка
- Веб-панель (отдельный React-раздел или страница), вход только для `admins` (Telegram
  Login + проверка id, или логин/пароль поверх). Показывает:
  - **Спрос**: топ броней и просмотров (в core.js уже есть `KV.track`, события шлём на бэкенд),
    что чаще заказывают, чего не хватает.
  - **Заказы**: список с составом, суммой, статусом, способом доставки.
  - **Клиенты**: имя, @username, телефон, привязка Telegram, их заказы и суммы.
  - Кнопки: запустить синк Sheets, отправить рассылку, отметить заказ выполненным.

## 4. Бот (стек и хостинг)

- Библиотека: **grammY** (Node, лёгкий) или **aiogram** (Python). Раз backend демо на Node,
  логично бот на grammY, общий код и деплой.
- Режим: **webhook** за nginx (надёжнее polling на VPS). Роут `/tg/webhook/<secret>`.
- Кнопка-меню = Web App на мини-апп (как сейчас хаб). Токен бота в секретах, не в репозитории.
- Команды: `/start` (регистрирует `bot_users`, opted_in=true), меню-кнопка открывает аппу,
  inline-кнопки для брони/заказа.

## 5. Деплой на VPS mcr

На mcr уже крутится MCR Planet. KatoVape ставим рядом, не мешая:
- Отдельный каталог `/opt/katovape`, отдельный `systemd`-юнит `katovape-bot.service` и
  `katovape-api.service`, свои порты (например 8791 API, 8792 webhook).
- nginx: отдельный server-блок на поддомен, например `shop.<домен>` или `katovape.<домен>`,
  TLS через тот же certbot. Проксирует API и `/tg/webhook`.
- Мини-апп статикой можно продолжать держать на GitHub Pages, а можно отдать с VPS тем же
  nginx (тогда всё на одном домене, проще с CORS и cookie). Реши, что удобнее.
- Секреты (`TELEGRAM_BOT_TOKEN`, ключ сервис-аккаунта Google, `STRIPE_SECRET`,
  `STRIPE_WEBHOOK_SECRET`) в `/opt/katovape/.env`, права 600, в git не кладём.
- БД: Postgres на VPS (или Supabase, если хочешь managed). rsync-деплой как у MCR, но помним
  урок: правки на сервере затираются деплоем, единственный источник — репозиторий.

## 6. Безопасность

- Токен бота, service-role, Stripe-секреты только на сервере.
- Подпись Telegram (`initData`, widget) проверяется бот-токеном на бэкенде (алгоритм уже
  написан в `supabase/functions/telegram-auth`).
- Админ-доступ по allowlist `admins.telegram_id` + серверная проверка на каждом запросе.
- Stripe webhook — проверка подписи. Пароли — bcrypt/scrypt. Данные клиентов отдаём только
  админам, обычный пользователь видит лишь своё.

## 7. Этапы

1. Перенести демо-бэкенд на Postgres, поднять API на VPS (users, orders, reservations).
2. Бот на grammY: /start, меню-кнопка, приём брони, webhook за nginx.
3. Синк Google Sheets → products, витрина читает БД.
4. Уведомления о поступлении по брони.
5. Админка: заказы, клиенты, спрос, кнопка рассылки.
6. Рассылки через бота.
7. Stripe Checkout + webhook, статусы заказов.

## 8. Что нужно от заказчика (вопросы)

- Оптовые цены: подтвердить пороги (1/3/5 шт) и что они у каждого товара свои.
- Аватар на сайте: демо кладёт фото в БД как data-URL; в прод хранить в Storage?
- Google Sheets: дать структуру колонок и доступ (сервис-аккаунт), один лист или по городам.
- Админы: список Telegram ID, кому можно в панель.
- Рассылки: аудитория = все, кто нажал /start, или ещё сегменты (по городу)?
- VPS: какой поддомен под KatoVape, мини-апп держим на GitHub Pages или переносим на VPS.
- Stripe: аккаунт есть, какие способы оплаты включить (карты, BLIK, Przelewy24).
